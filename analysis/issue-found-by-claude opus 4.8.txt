## The single most important finding

Your `/api/fix-vapi-tool` endpoint is broken, and it has been failing the **entire** time, every single time it's called:

```
Failed to fetch all assistants from Vapi: Cannot send a request, as the client has been closed.
INFO: ... "GET /api/fix-vapi-tool HTTP/1.1" 200 OK
```

This line appears over and over from Jun 25 through Jun 27. "Cannot send a request, as the client has been closed" is a Python `httpx` error. It means somewhere in your code the shared `httpx.AsyncClient` got `.aclose()`'d (closed), and then later code tries to reuse that same closed client. Once an `AsyncClient` is closed, every future request through it throws this exact error until the process restarts.

This is almost certainly the root of your tool problem. The tool creation/attachment flow and the "fix-vapi-tool" flow both talk to Vapi through that same httpx client. If the client is dead, your backend silently fails to create or sync tools on Vapi — but still returns `200 OK` because the error is being swallowed (caught and logged, not raised). That's why the dashboard shows "Missing tool" orphans and why `save_order` won't attach: your server thinks it succeeded, Vapi never got a valid request.

Notice the same failure also breaks `/api/calls`:

```
GET /api/calls HTTP/1.1" 500 Internal Server Error
... res = await client.get(f"{VAPI_BASE}/call", ...)
```

Same closed-client symptom, different endpoint.

## Why "it worked 3 days ago"

The log shows your service was last restarted **Jun 25 10:36** (pid 176599) and again **Jun 25 11:36** (pid 176909), and pid 176909 has been running ever since. The "client has been closed" errors start right after that and never stop. So a process that's been up for ~2 days is running with a permanently dead httpx client. A restart 3 days ago likely introduced a code change (or a shutdown/startup ordering bug) where the client gets closed prematurely — for example, an `async with httpx.AsyncClient() as client:` block that exits and closes the client, while a long-lived reference to it is reused on later requests. Or a FastAPI `shutdown`/lifespan handler closing the client that other code still holds.

## What to look at / do

First, find where the httpx client is created and closed. Look in your FastAPI code for:

```bash
cd ~/project/abdel-ai-voice-agent
grep -rn "AsyncClient\|aclose\|httpx" --include=*.py .
grep -rn "fix-vapi-tool\|fix_vapi_tool\|VAPI_BASE" --include=*.py .
```

You're looking for one of these patterns:
- A module-level `client = httpx.AsyncClient()` that some shutdown handler or `async with` closes, while other endpoints keep using it.
- A client created inside a function with `async with`, where a reference escapes and is reused later.

The robust fix is to stop sharing one long-lived client that can die. Either create a fresh client per request:

```python
async with httpx.AsyncClient(timeout=30) as client:
    res = await client.get(f"{VAPI_BASE}/tool", headers=headers)
```

or manage one client properly through FastAPI's lifespan and never `.aclose()` it except on real app shutdown.

Second, and this is important: your error handling is swallowing failures and returning 200. The fix-vapi-tool endpoint logs the error but responds `200 OK`. That's why you had no idea tools weren't being created. Make tool create/attach raise (or at least return non-200 + the Vapi response body) so failures are visible.

To confirm the diagnosis quickly, just restart the service and immediately try creating `save_order` again:

```bash
sudo systemctl restart abdel-ai-voice-agent.service
sudo journalctl -u abdel-ai-voice-agent.service -f
```

If tool creation works right after a fresh restart and then breaks again later, that confirms the client-lifecycle bug 100%.
